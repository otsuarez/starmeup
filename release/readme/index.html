rls based meta packages

meta packages are a great way of putting together a set of package with specific versions. They are useful for doing deployments, but should a rollback be required, an automatic procedure is out of the question.

RLS based deployments are just a set of packages without any particular dependency (other than being part of the same project). Traditionally deploys had been triggered by the creation of an RLS jira issue.

semver service

receives a set of variables:

  • component name (github repo name)
  • branch
  • sha

keeps track of major and minor version numbers for each component per environment

metadata:

  • timestamp the requests
  • who created it (authenticating requests or ip address)

workflow:

  • search the db to check whether the sha exists.
  • if exists, returns the patch number
  • if it doesn't exists, creates the entry and returns the patch number
  • the patch number will be incremental for the component's major.minor last number
